# Incident: Production `/auth/register` returns 403

**Date:** 2026-05-27 → 2026-05-28 (ongoing)  
**Status:** Open  
**Severity:** P0 — blocks all closed-beta user registrations  
**Affected:** All real users attempting to register on https://smart-apply.io

## Symptoms

1. User fills out the registration form with a valid invite code.
2. Browser POSTs to `https://api.smart-apply.io/api/v1/auth/register`.
3. Response: **HTTP 403 Forbidden**.
4. Frontend toast: **"Die Sicherheitsüberprüfung ist fehlgeschlagen. Bitte lade die Seite neu und versuche es erneut."** (mapped from `EBADCSRFTOKEN` in `apps/web/src/lib/error-messages.ts`).
5. DevTools also shows Cloudflare Turnstile noise (`Blocked script execution in 'about:blank'`, `PAT challenge 401`, `unsafe-eval` CSP warnings) — these are confirmed unrelated (Turnstile's internal sandbox); see "Red herrings" below.

## Evidence

### Request as captured in Network tab
- `Cookie: __Host-csrf=<...>` is **present**
- `X-CSRF-Token` header is **absent**
- Origin: `https://smart-apply.io`
- Status: 403

### Server logs (Fly)
```
POST /api/v1/auth/register - 403 [EBADCSRFTOKEN]
```

### Curl reproduction WORKS (server is healthy)
Manually fetching csrf-token, attaching cookie + header, then posting succeeds in passing CSRF + Turnstile (soft-fail) and reaches the invite-code validator, which correctly returns `INVITE_CODE_INVALID` for a fake code:
```
POST /api/v1/auth/register
{ "code":"INVITE_CODE_INVALID", "statusCode":403 }
```
This proves the backend pipeline is correct end-to-end. The bug is in the browser → API request itself.

## Hypotheses still open

1. **Service worker serves a stale `apps/web` bundle.** PR #495 reworked SW behaviour to require user-driven update; users may still be running an older bundle that calls `getCsrfToken()` synchronously without the new `fetchCsrfToken()` fallback. Even after Cmd-Shift-R, the SW could be serving the previous app shell.
2. **OpenNext / Cloudflare Workers cache delay.** A new deploy may take several minutes to propagate; users hitting cached HTML still ship the old JS chunks.
3. **CSRF cookie scoping mismatch.** `__Host-csrf` is set by the API at `api.smart-apply.io` with `Path=/`. The double-submit pattern requires the header value to match the cookie value at request time. If the browser sends the cookie but the JS reads `null` from in-memory state, the header is omitted. The new fallback in #510 SHOULD cover that — unless the running JS is the old bundle (see #1).
4. **Workers/edge cache of the `/auth/csrf-token` response.** Unlikely (cookie response should be marked private) but worth verifying.

## Next diagnostic steps

1. **Verify which JS bundle the browser is running**. Open DevTools → Sources → `12naq5q4mg__d.js` (chunk seen in user trace) and confirm whether the `apiRequest` function contains the new `await fetchCsrfToken()` block. If not, the deploy hasn't reached the user yet (SW cache or CF cache).
2. **Manually unregister the service worker** in DevTools → Application → Service Workers → Unregister → reload twice.
3. **Force-reset the chunk cache** by adding a `?v=` query string to the page URL or using a different browser / private window without the SW.
4. **Trace network in incognito with SW disabled**. If registration succeeds there, the fix is correct and the only remaining problem is rolling out the new bundle past stale caches.
5. **Diff `apiRequest`** between commit `34b0971` (v3.4.5) and `HEAD` (post #510) by sourcemap to confirm the bundle on prod actually contains the fix.
6. **Consider invalidating the Cloudflare Worker cache** via wrangler or the dashboard purge.
7. **Verify `__Host-csrf` cookie attributes** in browser DevTools → Application → Cookies. If `SameSite` or `Path` is unexpected, the cookie may not reach the API at all.
8. **Add request-side logging** behind a flag: send the in-memory token state with each request to a debug endpoint to confirm whether the new code path is actually running in real browsers.

## Red herrings (verified unrelated)

- **CSP `unsafe-eval` warnings** → only emitted by Cloudflare's Turnstile widget inside its own sandboxed `about:blank` iframe. Our CSP already allows `unsafe-eval` for `script-src 'self'`. Confirmed via:
  ```
  curl -sI https://smart-apply.io/register | grep content-security-policy
  ```
  shows: `script-src 'self' 'unsafe-eval' 'unsafe-inline' https://challenges.cloudflare.com ...`
- **"Blocked script execution in 'about:blank'"** → same source. Turnstile uses sandboxed sub-iframes; the noise is normal.
- **PAT challenge 401** → Cloudflare Private Access Token probe; normal pre-flight.

## Workaround tried (does not work for end users)

- `localStorage.clear()` + cookie deletion + reload → did not fix symptom for real users.
- Hard refresh (Cmd-Shift-R) → did not fix symptom either; service worker likely keeps serving the old bundle.

## Decision points

- **Should we temporarily disable CSRF in production** (`ENABLE_CSRF=false`) to unblock beta signups while we trace the bundle delivery issue? Trade-off: removes the double-submit defense for ~24 h; risk is low given the invite-code gate already blocks unauthorized signups.
- **Should we add a kill-switch for the service worker** so the next deploy can force-unregister it across all clients?
